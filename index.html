Body text: The quick brown fox jumps over the lazy dog. Use for paragraphs and default content.

Caption text: Supporting labels, hints, and metadata. Smaller and muted.

Code 
<h1 class="text-h1">Heading 1</h1>
<p class="text-body">Body text.</p>
<p class="text-caption">Caption text.</p>

Accessibility (WCAG)

  • Ensure text meets 4.5:1 contrast ratio (AA) on its background.
  • Use semantic headings in order (h1 then h2, etc.).
  • Relative units (rem) support zoom and user font-size preferences.
Do
Dark text on light background. Good contrast.
Don't
Gray text on light background. Poor contrast.

Accessibility (WCAG)

  • Do not rely on color alone to convey meaning.
  • Ensure sufficient contrast for any text on each swatch.
  • Document which color combinations are approved for text and UI.
Do
Use text + color: e.g. "Primary" label on a colored swatch so meaning is clear without color alone.
Don't
Rely only on a colored dot or bar with no label; screen readers and color-blind users need text.

Buttons

Primary, secondary, and disabled variants. Minimum tap target on small screens.

Example
Code 
<button type="button" class="btn btn--primary">Primary</button>
<button type="button" class="btn btn--secondary">Secondary</button>
<button type="button" class="btn btn--primary" disabled>Disabled</button>

Accessibility (WCAG)

  • Minimum 44×44px touch target for interactive elements.
  • Visible focus indicator (focus ring) on keyboard focus.
  • Use <button> or <a> with a proper role; disabled buttons are not focusable.
Do

Large tap target, visible focus ring.

Don't

Too small; no focus style.

Default, focus, and error states. Each input has an associated label.

Accessibility (WCAG)

  • Every input must have a visible <label> associated via for and id.
  • Error state: use aria-invalid="true" and descriptive error text.
  • Visible focus style so users can see which field is focused.

Checkbox

Single and multiple choice with clear labels. Use a fieldset and legend for groups.

Example
Notify me by
Code 
<div class="checkbox-row">
  <input type="checkbox" id="terms" class="checkbox">
  <label for="terms">I agree to the terms</label>
</div>
<fieldset>
  <legend>Notify me by</legend>
  <div class="checkbox-group">...</div>
</fieldset>

Accessibility (WCAG)

  • Every checkbox has a visible label associated via for and id.
  • Use fieldset and legend for checkbox groups.
  • Visible focus indicator; adequate touch target size.
Do
Label each checkbox; group related options in a fieldset with a legend.
Don't
Use unlabeled checkboxes or multiple groups without a legend.

Radio

Single-choice groups. Use one radio group per question and a fieldset with legend.

Example
Delivery method
Code 
<fieldset>
  <legend>Delivery method</legend>
  <div class="radio-group">
    <div class="radio-row">
      <input type="radio" id="opt1" class="radio" name="delivery" value="standard">
      <label for="opt1">Standard</label>
    </div>
  </div>
</fieldset>

Accessibility (WCAG)

  • Every radio has a visible label; group related options in a fieldset with legend.
  • Use the same name for all options in one group.
  • Visible focus indicator; adequate touch target size.
Do
One fieldset per question; one name per group; every option labeled.
Don't
Ungrouped radios, missing labels, or multiple names in one visual group.

Select

Native dropdown styled to match the design system. Use for a single choice from a list.

Example
Code 
<div class="input-group">
  <label for="country">Country</label>
  <select id="country" class="select">
    <option value="">Select a country</option>
    <option value="us">United States</option>
  </select>
</div>

Accessibility (WCAG)

  • Associate a visible label with the select via for and id.
  • Visible focus indicator; ensure sufficient contrast for options.
Do
Label the select so users know what they're choosing.
Don't
Use an unlabeled dropdown or rely only on placeholder text.

Textarea

Multi-line text input for long-form content. Same focus and error patterns as text input.

Example
This field is required.
Code 
<div class="input-group">
  <label for="message">Message</label>
  <textarea id="message" class="textarea" rows="4" placeholder="..."></textarea>
</div>

Accessibility (WCAG)

  • Every textarea has a visible label; use aria-invalid and error text when invalid.
  • Visible focus indicator; allow resize if it helps users (e.g. resize: vertical).
Do
Label the textarea; show clear error message when validation fails.
Don't
Rely only on placeholder as the label.

Toggle

On/off switch built with a checkbox. No JavaScript required; state is native.

Example
Code 
<div class="toggle-row">
  <input type="checkbox" id="notifications" class="toggle" role="switch">
  <label for="notifications">Enable notifications</label>
</div>

Accessibility (WCAG)

  • Use a visible label; role="switch" exposes the control to assistive tech.
  • Visible focus indicator; adequate touch target for the switch.
Do
Label the toggle so users know what it controls.
Don't
Use an unlabeled switch or rely on color alone for state.

Card

Container with shadow and padding. Use for dashboard tiles and content blocks.

Example

Card title

Card body content. Layout reflows on narrow viewports.

Card with actions

Use when a card needs primary and secondary actions.

Code 
<article class="card">
  <h3 class="card__title">Card title</h3>
  <p class="card__body">Card body content.</p>
  <div class="card__actions">
    <button type="button" class="btn btn--primary">Primary</button>
    <button type="button" class="btn btn--secondary">Secondary</button>
  </div>
</article>

Accessibility (WCAG)

  • Focus order should follow the visual content order.
  • If the whole card is clickable, use a single focusable element (e.g. link or button).
  • Ensure sufficient contrast for all text inside the card.
Do
One focusable element per card (e.g. a link wrapping the card or a single "View" button). Clear heading and contrast.
Don't
Multiple ungrouped links/buttons inside one card with no clear order, or low-contrast text.

Accessibility (WCAG)

  • Consistent spacing improves readability and reduces cognitive load.
  • Use adequate spacing between interactive elements to avoid accidental taps or clicks.
  • Avoid cramping controls; touch targets benefit from the spacing scale.
Do

Comfortable gap between buttons.

Don't

Cramped; easy to mis-tap.

Example application

A static mock of how components combine in a dashboard. All elements use this design system.

Example

Dashboard

Overview

Summary and key metrics. Built with card, card__title, and card__body.

Active

Recent activity

List or table content goes here. Uses the same card component.

Actions

Use card__actions for primary and secondary buttons.

Code 
<div class="example-app">
  <header class="example-app__bar">
    <h3 class="example-app__title">Dashboard</h3>
    <button class="btn btn--primary">New item</button>
  </header>
  <div class="example-app__main">
    <article class="card">
      <h3 class="card__title">Title</h3>
      <p class="card__body">...</p>
      <div class="card__actions">
        <button class="btn btn--primary">Save</button>
        <button class="btn btn--secondary">Cancel</button>
      </div>
    </article>
  </div>
</div>

Accessibility (WCAG)

  • In a real app, use landmarks (banner, main, navigation) and a logical heading order.
  • Ensure focus order follows the visual layout and that all interactive elements are keyboard accessible.
